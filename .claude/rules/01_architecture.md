# Архитектурные правила — Yummy (FastAPI + SQLite + vanilla JS)

> Адаптация популярного «модульного монолита» под реальный стек проекта.
> Проект НЕ на Node/Next/Prisma — правила про Zod/DTO-слои неприменимы буквально.

## Принципы

- Тонкие роут-хендлеры в `app/main.py`: валидация (Pydantic) + вызов слоя данных.
- Вся работа с БД — только через `Store` в `app/db.py`. Никакого `sqlite3` вне него.
- Внешние сервисы (Anthropic, Telegram) — в отдельных модулях (`ai.py`, `notify.py`)
  с **graceful degradation**: нет ключа → детерминированный фолбэк, не 500.
- Модели ответа (`app/models.py`) НИКОГДА не отдают pw_hash, токены, внутренний
  auth-state.
- Immutability по возможности: возвращать новые объекты, не мутировать переданные.

## Запрещено

- Сырые SQL-запросы вне `Store` (риск SQLi, дрейф схемы).
- Бизнес-правила остатка/выдачи вне `Store` (атомарность списания там).
- Доверять `order_id`/`partner_id` из тела запроса для проверки владения —
  только через `user_orders()` / код заказа (та же `WHERE user_id=?`).
- God-функции >50 строк, файлы >800 строк (текущий `index.html` — исключение,
  но новые модули не раздувать).
- Тяжёлые зависимости ради мелочи (проект на stdlib-подходе: PBKDF2 вместо
  passlib, JWT на stdlib вместо python-jose, httpx вместо SDK).

## Структура новой фичи (backend)

```
app/
├── main.py           # + эндпоинт: Pydantic-вход → Store-метод → response_model
├── db.py             # + метод Store: транзакция под self._lock
├── models.py         # + In/Out Pydantic-модели (секреты не наружу)
└── <feature>.py      # если внешний сервис — отдельный модуль с фолбэком
tests/
└── test_<area>.py    # тесты параллельно с кодом
```

## Frontend

- Один файл `app/static/index.html`. Новые строки → добавить KZ-перевод в
  `KZ`/`KZ_RX`. Демо-паритет → зеркалить в client-store и `make docs`.
- Компоновка на CSS-переменных из существующей палитры (крем/чернильный/оранжевый),
  Sora+Manrope. Не хардкодить цвета.
