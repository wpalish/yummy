# Workflow — алгоритм работы над Yummy

## Перед задачей

1. Прочитать `CLAUDE.md` (карта проекта) и относящиеся файлы.
2. Понять требование; если дыры — задать вопросы, не додумывать молча.
3. Для нетривиальной фичи — показать план перед реализацией.

## Во время

4. Реализовать по плану малыми шагами.
5. Тесты параллельно с кодом (TDD предпочтительно — см. skill `/tdd`).
6. Прогнать: `rm -f spasibox.db* && .venv/bin/python -m pytest -q`.
7. Если менялся фронт — `make docs` (с `YUMMY_PAGES_API_BASE=…` чтобы не сбросить
   витрину в демо) и зеркалить логику в client-store `docs/index.html`.
8. Живая проверка в браузере через preview-инструменты, а не «на глаз».

## После

9. Коммит по Conventional Commits: `feat:`/`fix:`/`refactor:`/`test:`/`docs:`/`chore:`.
   Тело — что и зачем; для фиксов — какой баг закрыт.
10. Пуш в `main` (репо github.com/wpalish/yummy).
11. **Render не автодеплоит** (public-URL подключение) — после пуша бэкенда
    нужен Manual Deploy на дашборде render.com.
12. Обновить память проекта (`~/.claude/.../project_spasibox.md`) плотным абзацем
    с хешем коммита — по устоявшейся конвенции.

## Git-безопасность

- Перед `checkout`/`reset`/`rm -rf` в репо — сначала `git status`, засташить/закоммитить.
- `.env` и токены — только локально, в git не попадают (проверять staged-дифф).
- Не пушить секреты: `git diff --cached | grep -i token` перед пушем при сомнении.
